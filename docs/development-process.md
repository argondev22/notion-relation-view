# 開発プロセス

本ドキュメントは、このプロジェクトの開発プロセスを定義する。
今後のプロジェクトでも再利用できるテンプレートとしても機能する。

## 概要

アジャイルをベースとした、スピードと構造のバランスを取るアプローチ。
判断の基準は従来のフェーズ区分ではなく、**後から変更するコストの大きさ**で分ける。

## フェーズ

### フェーズ1: プロダクト定義（最初に1回）

**プロダクト要件定義書（PRD）** を作成する。カバーする内容：

- 課題定義: 何の問題を、なぜ解決するのか
- ターゲットユーザー: 誰のためのプロダクトか
- ゴールと成功基準: 何をもって成功とするか
- 機能一覧と優先度（Must / Should / Could / Won't）
- 非機能要件（パフォーマンス、セキュリティ等）
- 制約と前提条件

**成果物:** `docs/requirements.md`

### フェーズ2: アーキテクチャ設計（最初に1回）

**後から変更するコストが高いこと**を先に決める：

- 技術スタックの選定と理由
- システム構成図（全体のコンポーネント構成）
- 認証・認可の方式
- データモデルの概要（主要エンティティとリレーション）
- インフラとデプロイ戦略

これらはシステム全体に影響するため、最初に決定する。

**成果物:** `docs/design.md`

### フェーズ3: スプリントサイクル（繰り返し）

各スプリントは少数の機能に集中する。スプリントごとに：

1. **機能設計** — 実装する機能の詳細を決める
   - API仕様（エンドポイント、リクエスト/レスポンス形式）
   - UI設計（画面構成、ユーザーフロー）
   - 詳細データモデル（テーブルスキーマ、インデックス）
   - 処理ロジックとエッジケース

2. **実装** — コードを書く

3. **テスト** — 動作を検証する

4. **振り返り** — プロセスを見直して改善する

**スプリントごとの成果物:** `docs/sprints/sprint-N/` に機能設計を格納

## 判断基準

| いつ決めるか | 何を決めるか | 理由 |
|-------------|-------------|------|
| 最初に（フェーズ1-2） | 後から変えにくいこと | DB、認証方式、全体アーキテクチャの変更はコストが大きい |
| スプリントごと（フェーズ3） | 後から変えやすいこと | 個別のAPIレスポンス、UI詳細、フィルタロジックはその機能だけに影響 |
