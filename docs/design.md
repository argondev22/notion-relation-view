# アーキテクチャ設計

## 1. 技術スタック

### フロントエンド

| 技術 | 用途 | 選定理由 |
|------|------|---------|
| React 18 | UIフレームワーク | コンポーネントベースの開発、エコシステムの充実 |
| TypeScript | 型安全性 | 開発時のエラー検出、コードの可読性向上 |
| Vite | ビルドツール | 高速なHMR（ホットリロード）、軽量 |
| React Router | ルーティング | SPAのページ遷移管理 |
| Axios | HTTPクライアント | APIリクエスト処理 |
| Vitest | テスト | Viteと統合されたテストフレームワーク |

グラフ描画ライブラリは実装時に選定する（D3.js、Cytoscape.js等の候補あり）。

### バックエンド

| 技術 | 用途 | 選定理由 |
|------|------|---------|
| FastAPI | Webフレームワーク | 高速、型ヒントによる自動バリデーション、OpenAPIドキュメント自動生成 |
| SQLAlchemy | ORM | Pythonの標準的なORM、柔軟なクエリ構築 |
| Alembic | DBマイグレーション | SQLAlchemyとの統合、バージョン管理されたマイグレーション |
| Pydantic | データバリデーション | FastAPIとの統合、型安全なリクエスト/レスポンス定義 |
| Redis | キャッシュ | Notion APIレスポンスのキャッシュ、レート制限対策 |

### インフラ

| 技術 | 用途 | 選定理由 |
|------|------|---------|
| Docker Compose | コンテナ管理 | 開発環境の統一、サービス間の依存管理 |
| PostgreSQL 16 | データベース | 信頼性、JSON型サポート、拡張性 |
| Redis 7 | キャッシュ | 高速なインメモリストア |

本番環境のデプロイ先は未定。MVPではローカル開発環境を優先する。

## 2. システム構成

```text
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│              │     │              │     │              │
│   ブラウザ    │────→│  フロントエンド  │────→│  バックエンド   │
│              │     │  (React)     │     │  (FastAPI)   │
│              │     │  :3000       │     │  :8000       │
└──────────────┘     └──────────────┘     └──────┬───────┘
                                                 │
                                    ┌────────────┼────────────┐
                                    │            │            │
                              ┌─────▼─────┐ ┌───▼────┐ ┌────▼─────┐
                              │ PostgreSQL │ │ Redis  │ │ Notion   │
                              │ :5432      │ │ :6379  │ │ API      │
                              └───────────┘ └────────┘ └──────────┘
```

### データの流れ

1. ブラウザ → フロントエンド: ユーザー操作
2. フロントエンド → バックエンド: API リクエスト (REST)
3. バックエンド → PostgreSQL: ユーザー情報、トークン、ビュー設定の永続化
4. バックエンド → Redis: Notion APIレスポンスのキャッシュ
5. バックエンド → Notion API: ページ・リレーションデータの取得

## 3. 認証方式

Google OIDC（OpenID Connect）を採用する。

```text
ブラウザ → Google認証画面 → Googleからコールバック → バックエンドでIDトークン検証 → JWT発行
```

- ログイン: Google OIDCでIDトークンを取得し、バックエンドで検証後にJWTを発行
- セッション管理: JWTをフロントエンドで保持し、APIリクエスト時にAuthorizationヘッダーで送信
- Notionトークン: ユーザーが入力したインテグレーショントークンをAES-256-GCMで暗号化して保存

## 4. データモデル概要

```text
┌──────────────┐       ┌──────────────────┐       ┌──────────────┐
│    users     │       │  notion_tokens   │       │    views     │
├──────────────┤       ├──────────────────┤       ├──────────────┤
│ id (PK)      │──┐    │ id (PK)          │       │ id (PK)      │
│ google_id    │  │    │ user_id (FK)     │←──┐   │ user_id (FK) │←──┐
│ email        │  │    │ encrypted_token  │   │   │ name         │   │
│ name         │  └───→│ created_at       │   │   │ config       │   │
│ created_at   │       │ updated_at       │   │   │ created_at   │   │
│ updated_at   │       └──────────────────┘   │   │ updated_at   │   │
└──────────────┘                              │   └──────────────┘   │
       │                                      │                      │
       └──────────────────────────────────────┴──────────────────────┘
                          1:1                         1:N
```

- **users**: Google認証で作成されるユーザー情報
- **notion_tokens**: ユーザーごとのNotionインテグレーショントークン（暗号化保存、1:1）
- **views**: ユーザーが保存したグラフビュー設定（Could Have機能、1:N）

## 5. バックエンドのレイヤー構成

```text
┌─────────────────────────────────┐
│  API Layer (api/)               │  ← エンドポイント定義、リクエスト/レスポンス処理
├─────────────────────────────────┤
│  Service Layer (services/)      │  ← ビジネスロジック、ユースケースの実行
├─────────────────────────────────┤
│  Model Layer (models/)          │  ← データモデル定義、DB操作
├─────────────────────────────────┤
│  Core (core/)                   │  ← 設定、DB接続、共通ユーティリティ
└─────────────────────────────────┘
```

依存の方向: API → Service → Model → Core

- **API Layer**: FastAPIルーター。リクエストの受付とレスポンスの返却のみ
- **Service Layer**: ビジネスロジック。Notion APIとの通信、グラフデータ構築、キャッシュ管理
- **Model Layer**: SQLAlchemyモデル。DBスキーマとクエリ
- **Core**: アプリケーション設定、DB接続管理

## 6. フロントエンドの構成

```text
src/
├── components/    # UIコンポーネント（表示責務）
├── contexts/      # React Context（状態管理）
├── hooks/         # カスタムフック（ロジックの再利用）
├── services/      # API通信（バックエンドとの通信）
├── types/         # TypeScript型定義
└── utils/         # ユーティリティ関数
```

- 状態管理はReact Context APIを使用（小〜中規模のため、Reduxは不要）
- API通信はservices/に集約し、コンポーネントから直接APIを呼ばない

## 7. テスト方針

| テスト種別 | 対象 | ツール | タイミング |
|-----------|------|--------|-----------|
| ユニットテスト | 関数、クラス、コンポーネント単体 | pytest / Vitest | 実装と同時 |
| 結合テスト | API エンドポイント、DB操作 | pytest + テストDB | 機能完成時 |
| E2Eテスト | ユーザーフロー全体 | 未定（Playwright等） | スプリント終了時 |

- 実装と同時にテストを書く（TDDではなく、実装後すぐにテスト作成）
- PRDの受入基準をテストケースの基準とする
